Who is PABLO

PABLO is C++/MPI library for managing parallel linear octree/quadtree. It has been developed at Optimad Engineering srl and it will be released under the GNU Lesser General Public License. The aim of the project is to provide users with a tool to manage parallel adaptive grid of quadrilaterals or hexahedra without the burden of implementing MPI instructions in their applications code. Indeed, in PABLO they are implemented transparent MPI communications for both grid and user data; even the dynamical load-balancing for both grid and user data is possible and easy to perform with simple calls of methods of the library. PABLO allows adaptive mesh refinement by generating non-conforming grid with hanging nodes. One of main feature of the library is the low memory consuming in the basic configuration: ~30B per octant in 3D. Useful additional features are: the possibility to impose 2:1 balancing between octants and the option of storing the element intersections.

Tutorials

Some tutorials on the use of PABLO library are described here below. This web page is not an user guide or a programmer manual, but only a brief description of the results obtained by running some examples. All the reults here illustrated are obtained from the test files in the repository. Here no code is shown, but one can follow the description of the tutorials by reading comments and code in the test files.

Tutorial Test 0

The first file test0.cpp is a simple tutorial about the instantation of octree objects. In this example two 2-dimensional objects of class Para_Tree are declared. The first one by using the default constructor, that builds an octree with one octant with size 1 and coordinates of node 0 in the origin. The second octree is instantiated by custom constructor, passing as input parameter the coordinates of node 0 equal to (10,20,0) and the size of the octant equal to 250. Note that PABLO works in a logical domain (maximum size of the first ancestor octant in logical domain equal to 2m, with m=maximum refinement level allowed; while minimum size of finest octant equal to 1), the transformation from the logical domain to the physical one has to be implemented by the user in a particular class (Class_Map); by default a simple translation/scaling is supplied with the repository. The two octrees are saved in logical coordinates and in physical ones. The figure below shows both the octrees in logical domain.

Note that the two octants are exactly overlapped, in effect they have the same coordinates and size in the logical domain. In the next figure the same octrees are displayed in the physical domain.

Here the two octants are highly different in size (1/250), octant with size 1 in blue and octant with size 250 in red, and for the coordinates of their node 0.

Tutorial Test 1

In this test (test1.cpp) one octree is first refined globally one time, then is refined iteratively marking to be refined (i.e. marker=1) all the octants with the center inside a circle. At the end of the test one global refinement is performed again. The results is shown in the video below, starting from the ancestor octant up to the last global refinement.

Note that in the right half-side of the domain the 2:1 balance is deactivated. Moreover in the lower part of the domain simple consequential refinements are achieved, while in the upper part a while loop is performed in order to refine geometrically even new octant obtained by 2:1 balancing potentially inside the area of the circle.

Tutorial Test 2

The video below displays the result obtained by test2.cpp. This Test is analogous to the Test 1 with the 2:1 balance active for the whole domain and only simple adapting performed. The main difference is that the geometrically condition to set marker=1 is not imposed if the center of an octant lies in the circle but if at least one of its nodes is inthe area of the circle.

In the video can be noted as the refinements are smoother and the geomtry is better captured. This because no-one octant is not refined after its born in 2:1 balancing procedure, but these octants are already refined by using the condtion on nodes rather than on centers.

Tutorial Test 3

The test3 starts from the end point of the previous test. This tutorial involves the use of the adapting method to caorse the mes in an area of the domain. In particular a smaller circle is defined and the marker of every octant that lies inside is set equal to -1 (i.e. to be coarsened) and the 2:1 balance deactivated. This procedure is repeated 5 times as it can be seen in the video below.

Note that a global coarsening of one level is performed before the adaptive procedure in the smaller circle.
